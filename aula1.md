O Tester Ágil é o profissional responsável por ajudar o time a garantir a qualidade do software durante todo o processo de desenvolvimento. Diferente do modelo tradicional, onde o teste acontece apenas no final, no método ágil o teste acontece **desde o início e de forma contínua**.

Algumas características importantes de um Tester Ágil são:

- **Participação ativa no time:** trabalha junto com desenvolvedores, PO e demais membros da equipe.
- **Mentalidade de qualidade:** não pensa apenas em encontrar bugs, mas em evitar que eles aconteçam.
- **Comunicação constante:** participa de reuniões e discute melhorias com o time.
- **Automação de testes:** sempre que possível, cria testes automatizados para facilitar validações futuras.
- **Pensamento crítico:** analisa cenários diferentes de uso para identificar possíveis problemas.
- **Colaboração:** ajuda o time a melhorar processos e práticas relacionadas à qualidade.

O Tester Ágil não trabalha isolado. Ele faz parte do time e contribui para que todos tenham responsabilidade pela qualidade do produto.

---

# Em quais momentos o Teste Ágil aparece no desenvolvimento?

No desenvolvimento ágil, o teste **não é uma etapa separada**, ele acontece durante toda a Sprint.

### Planejamento da Sprint
Durante o planejamento, o tester ajuda a analisar as histórias de usuário e definir **critérios de aceitação**. Isso ajuda o time a entender exatamente o que precisa ser entregue.

**Exemplo:**  
Se a história é sobre login no sistema, já podem ser definidos critérios como:
- usuário válido deve conseguir acessar
- senha incorreta deve mostrar mensagem de erro

---

### Durante o Desenvolvimento
Enquanto o sistema está sendo desenvolvido, o tester já começa a pensar e criar cenários de teste.

Isso pode incluir:
- testes manuais
- criação de testes automatizados
- análise de possíveis falhas no comportamento do sistema

O objetivo é encontrar problemas **o mais cedo possível**.

---

### Durante a Validação da Sprint
Antes de entregar a funcionalidade, o time valida se tudo está funcionando de acordo com os critérios definidos.

O tester ajuda a verificar:
- se a funcionalidade funciona corretamente
- se não quebrou outras partes do sistema
- se atende às regras de negócio

---

### Retrospectiva
O tester também pode contribuir com melhorias no processo de desenvolvimento e qualidade do software.

Por exemplo:
- sugerir novos testes
- melhorar automação
- ajustar critérios de qualidade

# Code Review

## Quando um Code Review faz sentido?

O **Code Review** é a prática de revisar o código feito por outro desenvolvedor antes de ele ser integrado ao projeto. Isso ajuda a manter a qualidade do software.

Algumas situações em que ele faz muito sentido são:

### 1. Melhorar a qualidade do código
Durante a revisão, é possível identificar:
- bugs
- problemas de lógica
- código desnecessário
- possíveis melhorias

Isso evita problemas futuros no sistema.

---

### 2. Manter padrões do projeto
O Code Review ajuda a garantir que todos os desenvolvedores sigam:

- padrões de código
- boas práticas
- organização do projeto

Isso deixa o código mais fácil de entender e manter.

---

### 3. Compartilhar conhecimento
Quando alguém revisa o código de outra pessoa, acaba aprendendo novas formas de resolver problemas.

Isso ajuda o time a crescer tecnicamente e evita que apenas uma pessoa conheça determinada parte do sistema.

---

# Quando o Code Review deixa de fazer sentido?

Apesar de ser uma prática muito importante, ele pode perder o sentido em algumas situações.

### 1. Quando vira crítica pessoal
Se a revisão vira discussão ou ataque pessoal, o objetivo de melhorar o código se perde.

O foco deve ser sempre **no código, nunca na pessoa**.

---

### 2. Quando atrasa muito o desenvolvimento
Se a revisão demora demais ou fica travada por detalhes muito pequenos, ela pode atrasar a entrega da Sprint.

O ideal é focar no que realmente impacta a qualidade do software.

---

### 3. Quando é feito sem atenção
Se alguém aprova o código apenas por obrigação, sem realmente revisar, o Code Review perde totalmente o valor.

Nesse caso ele vira apenas um processo burocrático.

---

# Minha opinião sobre Code Review

Na minha opinião, o Code Review é uma prática muito importante dentro de equipes de desenvolvimento. Ele ajuda a melhorar a qualidade do software, evita erros e ainda promove aprendizado entre os membros do time.

Porém, para funcionar bem, ele precisa ser feito com **respeito, foco na melhoria do código e colaboração entre o time**. Quando vira apenas burocracia ou disputa de ego, deixa de cumprir seu verdadeiro objetivo.
